/**
 * # Reminder Scheduler Job-Runner Contract
 *
 * ## Overview
 *
 * The reminder scheduler implements a distributed job-runner pattern using Redis-based
 * deduplication and in-process autoscaling. This document specifies the correctness
 * properties and concurrency guarantees that form the basis of the implementation.
 *
 * ## Guarantee 1: Mutual Exclusion via Distributed Lock
 *
 * **Statement:** For any reminder R with (id, triggerAt), exactly one worker globally
 * will deliver R, even with unlimited concurrent workers.
 *
 * **Mechanism:**
 * - Each (id, triggerAt) pair maps to a Redis key: `reminder:dedup:{id}:{triggerAt}`
 * - On processing, a worker atomically executes: `SET key "1" EX 25h NX`
 * - Only one worker sees "OK" (lock acquired); others see nil (locked)
 * - The 25-hour TTL ensures eventual recovery if worker crashes
 *
 * **Implication for Duplicate Prevention:**
 * - Worker A and B fetch the same due reminder
 * - A: `SET ... NX` → "OK" (continues)
 * - B: `SET ... NX` → nil (skips, increments "skipped" metric)
 * - Even if 100 workers fetch the same reminder, only 1 delivers
 *
 * **Code Path:**
 * ```typescript
 * const claimed = await claimDelivery(reminder.id, reminder.triggerAt);
 * if (!claimed) {
 *   reminderMetrics.increment("skipped");
 *   continue; // Skip delivery
 * }
 * ```
 *
 * ## Guarantee 2: Idempotent State Transitions
 *
 * **Statement:** Re-executing a delivery operation (same reminder, same worker) after
 * a preemption or network partition produces the same final state.\n *\n * **Mechanism:**\n * - Delivery → `await repository.markSent(reminder.id, now)`\n * - Failure → `await repository.recordAttempt(reminder.id, now)`\n * - Both operations check current state and are side-effect free after first application\n * - Timestamps prevent ambiguity (using \"now\" snapshot, not current time)\n *\n * **Why This Matters:**\n * - Worker A claims lock, begins delivery, then crashes\n * - Lock expires; Worker B claims and completes delivery\n * - If A resumes (unlikely but possible), markSent sees reminder already sent → no error\n *\n * **Practical Impact:**\n * - No need for distributed consensus beyond the lock\n * - Crash recovery is automatic (no manual intervention)\n *\n * ## Guarantee 3: Time-Bound Consistency Window\n *\n * **Statement:** Lock TTL bounds the window in which a crashed worker's work can be\n * retried by another worker.\n *\n * **Guarantee Details:**\n * - Lock TTL = 25 hours\n * - If worker A crashes while holding lock, lock auto-releases after 25h\n * - Worker B can then claim lock and retry\n * - Within 25h, only A can retry (has lock)\n * - After 25h, only B can retry (A's lock expired)\n * - Window is exclusive: no overlap where both could claim\n *\n * **TTL Justification:**\n * - 25h > typical max retry duration (~3 attempts over ~10 minutes)\n * - Covers entire lifecycle of a single reminder processing attempt\n * - Larger TTL = safer, but slower recovery\n * - Smaller TTL = faster recovery, but risk of double-delivery during burst crashes\n *\n * ## Guarantee 4: Snapshot Consistency for Tick Alignment\n *\n * **Statement:** Within a single tick (call to processReminders), all reminders are\n * evaluated against the same \"now\" timestamp, preventing skew across chunks.\n *\n * **Mechanism:**\n * ```typescript\n * export async function processReminders(\n *   options: ProcessRemindersOptions & { reminders?: Reminder[] } = {}\n * ) {\n *   const now = options.now ?? Date.now();  // ← Single snapshot\n *   const dueReminders = options.reminders ?? (await repository.getDueReminders(now));\n *\n *   for (const reminder of dueReminders) {\n *     // All reminders processed with same \"now\"\n *   }\n * }\n * ```\n *\n * **Why This Matters:**\n * - Long-running tick (multiple chunks) doesn't drift\n * - All chunks use original \"now\" snapshot\n * - \"Fire-at-T\" reminders always fire in the tick when real time ≥ T\n * - Prevents off-by-one errors in next-fire calculations\n *\n * ## Guarantee 5: Crash Recovery Without Manual Intervention\n *\n * **Statement:** If a worker crashes mid-delivery, the system automatically recovers\n * without administrative action and without duplicate deliveries.\n *\n * **Scenario:**\n * ```\n * t=0:   Worker A claims lock for reminder R\n * t=10:  Worker A begins delivery\n * t=15:  Network partition; Worker A unreachable\n * t=20:  Worker A realizes error; attempts to release lock (fails; can't reach Redis)\n * \n * t=90000h: Lock TTL expires (25 hours)\n * t=90001h: Worker B fetches R as due\n * t=90001h: Worker B claims lock (now available)\n * t=90001h: Worker B delivers R successfully\n * ```\n *\n * **Correctness Argument:**\n * - Window [0, 90000h]: Only A can deliver (holds lock)\n *   - If A was genuinely working: great, it completes\n *   - If A was truly crashed: no delivery happened, no side-effect\n * - Window [90000h, ∞): Only B can deliver (A's lock gone, B has new lock)\n *   - Safe because A's delivery attempt is \"ancient\" (25h old)\n * - No overlap where both could deliver\n *\n * ## Guarantee 6: Concurrent Workers Converge to Same State\n *\n * **Statement:** Whether 1 or 100 workers process the same reminders, the final\n * repository state is identical.\n *\n * **Mechanism:**\n * - Dedup ensures only one completes delivery\n * - Others see \"skipped\" and don't modify repository\n * - `markSent`, `recordAttempt` are deterministic (same inputs → same output)\n *\n * **Implication:**\n * - Autoscaler can change concurrency dynamically\n * - More workers = lower latency, same final state\n * - Fewer workers = higher latency, same final state\n *\n * ## Edge Cases & Mitigations\n *\n * ### Edge Case 1: Stale Lock Recovery (Double-Emit Risk)\n *\n * **Scenario:**\n * ```\n * t=0:   Worker A: claim() → true, begins delivery\n * t=10:  Worker A: preempted by OS (e.g., CPU burst limit)\n * t=15:  Worker B: claim() → false (A's lock still active)\n * \n * t=25h: Worker A: resumes, completes delivery, calls markSent()\n * t=25h: Lock TTL expires\n * t=25h+1: Worker B: claim() → true, begins delivery\n * t=25h+1: Worker B: calls markSent() (idempotent, no error)\n * ```\n *\n * **Mitigation:**\n * - `markSent` is idempotent: calling twice = same state\n * - Both A and B see reminder as \"sent\"; no metrics double-count\n * - Key: Repository layer must tolerate repeated markSent calls\n *\n * ### Edge Case 2: Network Partition Before Lock Release\n *\n * **Scenario:**\n * ```\n * Worker A claims lock in Redis\n * Network partition prevents A from seeing Redis again\n * Worker A thinks it still has lock; actually it's still held but unreachable\n * After 25h, lock expires; Worker B claims and retries\n * ```\n *\n * **Mitigation:**\n * - 25h TTL window prevents A from re-claiming after B\n * - B's delivery + state change is final\n * - A's delivery (if it completes via different network) is idempotent\n * - Repository must handle timestamp comparisons carefully\n *\n * ### Edge Case 3: Clock Skew Between Workers\n *\n * **Scenario:**\n * ```\n * Worker A: system clock is 10s behind B\n * A: now = 1000 (actually 1010 globally)\n * B: now = 1010\n * Both fetch \"due\" reminders at their local \"now\"\n * A's snapshot includes reminders due at [0, 1000]\n * B's snapshot includes reminders due at [0, 1010]\n * ```\n *\n * **Mitigation:**\n * - Dedup prevents double-delivery even if snapshots differ\n * - Reminder will be processed in next tick by B if A misses\n * - Recommendation: Sync worker clocks (NTP, SystemD-timesyncd)\n * - Redis timestamp for lock/dedup is authoritative (Redis server time)\n *\n * ## Testing Strategy\n *\n * The test suite (`scheduler-job-runner.test.ts`) covers:\n *\n * 1. **Concurrency Test**: 5+ workers claim same reminder; only 1 delivers\n * 2. **Crash Recovery Test**: Lock expires; new worker retries without duplicate\n * 3. **Drift Test**: Long-running tick maintains snapshot consistency\n * 4. **Stale Lock Test**: Promoted worker after TTL can't double-emit\n * 5. **Variable Concurrency**: 1→N workers with proper metrics\n * 6. **Concurrent Failures**: Multiple failures don't double-count attempts\n * 7. **TTL Boundaries**: Reminders behave correctly at lock TTL transitions\n * 8. **Autoscaler Dedup**: Concurrency changes don't break dedup\n * 9. **Complex Crash Scenario**: Multiple retries across TTL boundary\n * 10. **Monotonicity**: Event ordering preserved despite concurrency\n *\n * All tests use `jest.useFakeTimers()` to:\n * - Avoid flaky timing dependencies\n * - Simulate TTL expiry instantly\n * - Control concurrent execution order\n * - Verify exact behavior under pathological timing\n *\n * ## Implementation Checklist\n *\n * - [x] Lock mechanism: Redis SET ...
